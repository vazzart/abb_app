# Android Bridge Bot — Архитектура и план реализации

## Обзор

Система читает SMS с Android-смартфона, подключённого по USB, сохраняет их в локальную БД и доставляет в Telegram. Локальная БД служит буфером надёжности: если доставка не прошла — сообщение будет отправлено повторно или через альтернативный канал.

---

## Технический стек

| Компонент | Технология | Обоснование |
|---|---|---|
| Язык | Go | нативная конкурентность, хорошая поддержка системного уровня |
| USB/ADB | `go-adb` (`github.com/zach-klippenstein/goadb`) | Go-обёртка над протоколом ADB |
| База данных | SQLite (`modernc.org/sqlite`) | без CGO, zero-deploy, достаточно для локального хранилища |
| Telegram | `github.com/go-telegram-bot-api/telegram-bot-api/v5` | стабильная библиотека для Bot API |
| SOCKS5 прокси | `golang.org/x/net/proxy` | stdlib-совместимый диалер; пробрасывается в `http.Transport` |
| Конфиг | `github.com/spf13/viper` + `.env` / YAML | читаемый конфиг без перекомпиляции |
| Логи | `go.uber.org/zap` | структурированные логи, легко парсить |

**Предусловие на телефоне:** включён режим разработчика + USB-отладка (ADB debugging).

### Компоненты

#### USB Watcher
- Отслеживает подключение/отключение ADB-устройства.
- При подключении — запускает SMS Poller.
- При отключении — останавливает поллер, ждёт повторного подключения.

#### SMS Poller
- Каждые N секунд выполняет `adb shell content query --uri content://sms/inbox`.
- Сравнивает полученные `_id` с уже сохранёнными в БД.
- Новые сообщения передаёт в Normalizer.

#### Normalizer
- Приводит сырые поля ADB к внутренней структуре `Message`.
- Опционально фильтрует по номеру, ключевому слову и т.д.
- Сохраняет сообщение в таблицу `messages` (статус `pending`).

#### Dispatcher
- Читает из `outbox` (view/таблица сообщений со статусом `pending`).
- Вызывает все зарегистрированные Sender-ы.
- Записывает результат (success / failed + reason) в `delivery_log`.

#### TelegramSender
- Реализует интерфейс `Sender`.
- Отправляет сообщение конкретному `chat_id`.
- При ошибке API возвращает ошибку — Dispatcher помечает запись как `failed`.
- Опционально использует SOCKS5-прокси: создаёт кастомный `http.Transport` с `proxy.Dialer` и передаёт его в HTTP-клиент бота.
- Поддерживает аутентификацию прокси (логин + пароль) через конфиг; если прокси не задан — используется прямое подключение.

#### Retry Worker
- Тикер (например, каждые 5 минут).
- Берёт сообщения со статусом `failed` и `attempts < max_attempts`.
- Передаёт их обратно в Dispatcher.

## Схема базы данных

```sql
-- Входящие SMS
CREATE TABLE messages (
    id           INTEGER PRIMARY KEY,
    android_id   TEXT UNIQUE,          -- _id из content://sms
    address      TEXT NOT NULL,        -- номер телефона
    body         TEXT NOT NULL,
    body_edited  TEXT,                 -- если пользователь отредактировал перед отправкой
    received_at  DATETIME NOT NULL,
    created_at   DATETIME DEFAULT CURRENT_TIMESTAMP,
    status       TEXT DEFAULT 'pending' -- pending | sent | skipped
);

-- Очередь доставки по каналам
CREATE TABLE outbox (
    id          INTEGER PRIMARY KEY,
    message_id  INTEGER NOT NULL REFERENCES messages(id),
    channel     TEXT NOT NULL,         -- 'telegram', 'email', ...
    status      TEXT DEFAULT 'pending',-- pending | sent | failed | cancelled
    attempts    INTEGER DEFAULT 0,
    last_error  TEXT,
    scheduled_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    sent_at     DATETIME
);

-- Полная история попыток доставки
CREATE TABLE delivery_log (
    id         INTEGER PRIMARY KEY,
    outbox_id  INTEGER NOT NULL REFERENCES outbox(id),
    attempted_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    success    BOOLEAN NOT NULL,
    error      TEXT
);
```

## Интерфейс Sender

```go
// Sender — контракт для любого канала доставки.
// Каждый новый канал реализует этот интерфейс и регистрируется в Dispatcher.
type Sender interface {
    Name() string
    Send(ctx context.Context, msg Message) error
}
```

## Этапы реализации

### Этап 1 — Скелет и ADB-подключение
**Цель:** убедиться, что можем читать SMS через ADB.

- [ ] Инициализировать Go-модуль, добавить зависимости.
- [ ] Реализовать `adb.Watcher`: обнаружение подключения устройства.
- [ ] Реализовать `adb.SMSPoller`: выполнить `content query`, распарсить строки в `Message`.
- [ ] Консольный вывод прочитанных SMS — проверка корректности парсинга.

**Результат:** приложение печатает SMS в терминал при подключении телефона.

---

### Этап 2 — Локальная БД
**Цель:** надёжное хранение всех входящих сообщений.

- [ ] Подключить SQLite, реализовать миграции (`db/schema.go`).
- [ ] CRUD для `messages` и `outbox`.
- [ ] При обнаружении новых SMS — сохранять в `messages` + создавать запись в `outbox` (канал `telegram`).
- [ ] Дедупликация по `android_id`.

**Результат:** повторный запуск не создаёт дубликатов; история SMS сохраняется локально.

---

### Этап 3 — Telegram-доставка
**Цель:** отправлять SMS одному получателю в Telegram.

- [ ] Реализовать `sender.TelegramSender` (интерфейс `Sender`).
- [ ] Реализовать `Dispatcher`: читает `outbox` со статусом `pending`, вызывает Sender, обновляет статус.
- [ ] Конфигурация: `TELEGRAM_TOKEN`, `TELEGRAM_CHAT_ID`.
- [ ] Форматирование сообщения: `[+7XXXXXXXXXX] текст смс`.

**Результат:** входящие SMS появляются в Telegram-чате.

---

### Этап 4 — Retry-механизм
**Цель:** не терять сообщения при временных сбоях сети / Telegram API.

- [ ] Реализовать `dispatcher.RetryWorker` (тикер).
- [ ] Политика: экспоненциальный backoff, максимум `max_attempts` (настраивается).
- [ ] После исчерпания попыток — статус `failed`, алерт в лог (и/или системное уведомление).
- [ ] Запись каждой попытки в `delivery_log`.

**Результат:** при перезапуске и временных сбоях сообщения доставляются без потерь.

---

### Этап 5 — Редактирование перед отправкой (опционально)
**Цель:** возможность изменить текст SMS до доставки.

- [ ] CLI-команда или простой HTTP-эндпоинт: `GET /messages?status=pending`, `PATCH /messages/{id}` (поле `body_edited`).
- [ ] Dispatcher использует `body_edited` если задан, иначе `body`.
- [ ] Возможность отменить доставку: `PATCH /outbox/{id}` → статус `cancelled`.

**Результат:** можно просмотреть очередь и отредактировать/отменить сообщение до отправки.

---

### Этап 6 — Второй канал доставки (пример: Email)
**Цель:** продемонстрировать расширяемость через интерфейс `Sender`.

- [ ] Реализовать `sender.EmailSender` (SMTP).
- [ ] Зарегистрировать в Dispatcher рядом с Telegram.
- [ ] Конфигурация: выбор каналов для конкретных номеров/масок.

**Результат:** архитектура подтверждает расширяемость без изменения ядра.

---

### Этап 7 — Продакшн-готовность
**Цель:** стабильная долгосрочная работа как системного сервиса.

- [ ] systemd unit-файл для автозапуска.
- [ ] Graceful shutdown (context cancellation).
- [ ] Мониторинг: метрики (Prometheus) или хотя бы счётчики в логах.
- [ ] Ротация логов.
- [ ] Документация: `README.md`, описание настройки ADB на телефоне.

### Этап 4 — Retry-механизм

| ID | Название | Шаги | Ожидаемый результат |
|---|---|---|---|
| T4-01 | Повтор после восстановления сети | Отключить сеть, дождаться статуса `failed`, включить | Retry Worker повторяет отправку; статус → `sent` |
| T4-02 | Экспоненциальный backoff | Наблюдать интервалы между попытками | Каждый следующий интервал длиннее предыдущего (×2) |
| T4-03 | Исчерпание попыток | `max_attempts=3`, Telegram недоступен постоянно | После 3 попыток статус → `failed` навсегда; в лог пишется предупреждение |
| T4-04 | Лог всех попыток | Одно сообщение, 2 неудачи + успех | В `delivery_log` 3 записи: 2 с `success=false`, 1 с `success=true` |
| T4-05 | Retry не дублирует успешные | Запустить Retry Worker вручную | Сообщения со статусом `sent` не отправляются повторно |
| T4-06 | Graceful shutdown во время retry | Послать SIGTERM в момент ожидания backoff | Приложение завершается чисто, незавершённая попытка не теряет статус |

### Этап 5 — Редактирование перед отправкой

| ID | Название | Шаги | Ожидаемый результат |
|---|---|---|---|
| T5-01 | Редактирование тела | `PATCH /messages/{id}` с `body_edited` | В Telegram уходит `body_edited`, а не оригинальный `body` |
| T5-02 | Оригинал не затёрт | После редактирования запросить запись | Поле `body` не изменилось, `body_edited` содержит новый текст |
| T5-03 | Отмена доставки | `PATCH /outbox/{id}` → `cancelled` | Dispatcher пропускает запись; повторный запуск её не трогает |
| T5-04 | Редактирование уже отправленного | `PATCH` для `outbox` со статусом `sent` | Ошибка 409 Conflict; изменение не применяется |
| T5-05 | Пустое `body_edited` | Передать `body_edited: ""` | Возвращается ошибка валидации; пустое тело не сохраняется |
| T5-06 | XSS / injection в теле | Передать `<script>alert(1)</script>` | Текст сохраняется как строка, не интерпретируется |

| T7-02 | Graceful shutdown | `systemctl stop abb` | Текущий poll/send завершается, БД закрывается без ошибок |

| T7-04 | Ротация логов | Запустить с большим объёмом логов | Старые файлы ротируются по размеру/дате, диск не переполняется |
| T7-05 | Метрики | Запросить `/metrics` (Prometheus) | Счётчики `sms_received_total`, `sms_sent_total`, `sms_failed_total` присутствуют |

### Сквозные (end-to-end) тест-кейсы

| ID | Название | Описание |
|---|---|---|
| E2E-01 | Полный путь: SMS → Telegram | Отправить реальное SMS на телефон, убедиться что оно пришло в Telegram ≤ poll_interval + 5s |
| E2E-02 | Сценарий разрыва | SMS пришло → сеть упала → сеть восстановилась → SMS доставлено повторно |
| E2E-03 | Массовый импорт | 100 SMS в inbox при первом запуске → все в БД, все доставлены, нет дублей |
| E2E-04 | Переподключение телефона | SMS пришло пока телефон был отключён → при подключении импортировалось и доставлено |

## Риски и ограничения

| Риск | Митигация |
|---|---|
| ADB требует ручного подтверждения на телефоне при первом подключении | документировать в README; Watcher повторяет попытки |
| Производитель может ограничить доступ к `content://sms` | тестировать на целевом устройстве на этапе 1 |
| Большое число SMS при первом запуске | на этапе 2 добавить параметр `import_since` — импортировать только с заданной даты |
| Telegram Bot API rate limit (30 msg/sec) | Dispatcher добавляет задержку между отправками |
| Телефон разряжается / перезагружается | Watcher обрабатывает reconnect, БД сохраняет состояние |
| SOCKS5 прокси недоступен — сообщения не доставляются | Retry Worker повторяет; если прокси мёртв насовсем — переключиться вручную или реализовать fallback на прямое соединение |
| Утечка учётных данных прокси из конфига | Хранить `username`/`password` в переменных окружения, не в YAML-файле |
